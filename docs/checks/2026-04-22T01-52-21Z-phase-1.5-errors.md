## [09] Real functional gaps discovered in dev tree (NOT cache-stale)

### 9.1 get_order_book — `depth` not normalised for KuCoin
- Tool accepts `depth: int Field(ge=1, le=200)`, passes raw to `CCXTProvider._fetch_order_book_raw` → `self._exchange.fetch_order_book(limit=depth)`.
- KuCoin allows **only** `limit ∈ {20, 100}`. Everything else → `kucoin fetchOrderBook() limit argument must be 20 or 100`.
- **Dev-tree fix:** snap depth per venue (e.g. `{20, 100}` for KuCoin) or raise `ValidationError` at the tool boundary with a venue-specific message.

### 9.2 get_trades — not implemented in either provider
- `CCXTProvider` overrides only `_fetch_{ticker,ohlcv,order_book}_raw`.
- `CoinGeckoProvider` overrides only `_fetch_ticker_raw` (it also documents the gap in its module docstring).
- `BaseProvider._fetch_trades_raw` raises `NotImplementedError("trades not implemented for this provider")`.
- The MCP tool `get_trades` is therefore **dead on arrival** for both venues.
- **Dev-tree fix:** implement `_fetch_trades_raw` for CCXTProvider (`await self._exchange.fetch_trades(symbol.native_symbol, since, limit)`) + normaliser via `CCXTAdapter.trades_to_domain`; add contract tests.

Both gaps are **real in the dev tree**; they would persist even after the marketplace cache is refreshed.

## [13] Bonus fix discovered during runtime

🔴 `cryptozavr://categories` raised `KeyError` → client saw "Error reading resource".

Root cause: CoinGecko `/coins/categories` returns `{id, name, market_cap, market_cap_change_24h, ...}` but `CategoryDTO.from_provider` reads `raw["category_id"]`. `CoinGeckoAdapter.categories_to_list` just copied dicts without normalisation. The `except Exception` in the resource only wraps the HTTP call; the DTO comprehension ran unprotected.

Fix: commit `111cde7 fix(infrastructure): map CoinGecko category id to category_id for DTO`. Adapter now copies `id` → `category_id` defensively (+2 tests). Live resource now returns 30 real categories with `market_cap_change_24h_pct`.
